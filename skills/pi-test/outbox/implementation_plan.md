# Implementation Plan

## 1. Set up the project
- Create a FastAPI project structure
- Add dependency management
- Add Dockerfile and docker-compose or Podman equivalent
- Configure PostgreSQL connection
- Add environment variables for database URL and app settings

## 2. Define the database schema
Create these initial tables:
- `users`
  - id
  - name
  - email
  - password_hash
  - role (`patient` or `admin`)
  - created_at
- `doctors`
  - id
  - name
  - specialty
  - created_at
- `availability_slots`
  - id
  - doctor_id
  - start_time
  - end_time
  - is_available
- `appointments`
  - id
  - patient_id
  - doctor_id
  - slot_id
  - status (`booked`, `cancelled`)
  - created_at
  - cancelled_at

## 3. Create basic project structure
Suggested folders:
- `app/main.py`
- `app/api/`
- `app/models/`
- `app/schemas/`
- `app/services/`
- `app/db/`
- `app/core/`

## 4. Implement authentication
For the prototype:
- Add patient registration endpoint
- Add login endpoint
- Use simple JWT bearer authentication or a very basic token-based approach
- Protect admin endpoints by role

## 5. Implement doctor endpoints
Build endpoints to:
- List doctors
- Search doctors by specialty
- Add a doctor (admin only)
- Update doctor availability (admin only)

## 6. Implement appointment endpoints
Build endpoints to:
- Book an appointment
- Cancel an appointment
- View a patient's appointment history
- View all bookings (admin only)

## 7. Add validation rules
Important checks:
- Email must be unique
- A slot must exist before booking
- A booked slot cannot be double-booked
- Patients can only cancel their own appointments
- Cancelled appointments should not appear as active bookings
- Admin-only endpoints must reject patient users

## 8. Seed sample data
Add seed data for:
- Sample doctors from the inbox
- Sample patients if useful for demo
- Example availability slots

## 9. Add error handling
Return clear errors for:
- Invalid login
- Missing records
- Duplicate email
- Booking conflicts
- Unauthorized or forbidden access
- Invalid request payloads

## 10. Add tests
Start with simple tests for:
- Register
- Login
- List/search doctors
- Book appointment
- Prevent double-booking
- Cancel appointment
- Admin add doctor

## 11. Add API documentation
- Use FastAPI autogenerated OpenAPI docs
- Add request/response examples
- Document auth usage

## 12. Prepare local run instructions
Include:
- How to start PostgreSQL
- How to run the API locally
- How to run with Docker or Podman
- How to seed demo data

## Recommended build order
1. Project setup
2. Database models
3. Registration/login
4. Doctor listing/search
5. Appointment booking/cancellation
6. Admin features
7. Tests and demo data

## Beginner-friendly implementation notes
- Keep business rules in service functions, not directly in route files
- Use Pydantic schemas for all request and response bodies
- Start simple before adding advanced features like reminders or calendar sync
